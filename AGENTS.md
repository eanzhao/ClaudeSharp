# Repo Notes

## Git Workflow

- 默认在完成改动并通过相关验证后，直接提交并推送到当前工作分支。
- 如果用户明确说了不要推、不要提交、先别动远端，按用户要求执行。
- 推送前不要把无关改动混进提交；如果工作区里有不属于本次任务的改动，先只提交本次任务相关文件。

## Architecture Guardrails

- 严格分层：`Aexon.Core` 放稳定语义、运行时编排和持久化边界；`Aexon.Commands` 放命令入口；`Aexon.Tools` 放工具实现；`Aexon.Cli` 只做宿主、参数解析和组合，不承载核心业务决策。
- 单一主链路：同一个能力只能有一条权威执行路径。新增 provider、tool loop、session 恢复、streaming 等能力时，优先接到现有主链路上，禁止再做一套平行实现。
- 读写职责分离：会改变会话、运行态或持久化数据的逻辑，必须走明确的命令或运行时流程；查询、展示、状态读取走只读路径，不能在读取时偷偷补业务语义。
- 状态归属清楚：跨消息、跨回合、跨会话要保留的事实状态，必须有明确拥有者和落盘位置；禁止把进程内字典、缓存或静态变量当成事实源。
- 边界语义收口到 Core：跨模块共享的 provider 选择、消息转换、会话存储、权限判断等共识，应该沉到 `Aexon.Core`，不要在 `Cli`、`Commands`、`Tools` 各写一份。
- 依赖方向保持单向：上层依赖抽象和稳定接口，不能为了省事让 `Core` 反向引用 `Cli` 细节，也不要让命令层直接耦合具体宿主实现。
- 强类型优先：影响控制流、协议兼容、持久化格式或用户可见行为的数据，优先建成明确类型；少用松散的 `Dictionary<string, object>`、魔法字符串和隐式约定。
- 命名先表达职责：类型、目录、命名空间优先说清“它负责什么”，少用 `Helper`、`Manager`、`Util` 这类兜底词；重复抽象如果没有独立语义，就合并或删除。
- 删除优于兼容空壳：迁移或重构完成后，旧实现、空转发、临时兼容层和无用依赖要及时清掉，不保留“也许以后还用得上”的死代码。
- 变更必须可验证：改主链路、存储格式、provider 适配、工具协议或命令行为时，至少补上对应测试；如果改变了约定或使用方式，也要同步更新 `README.md` 或 `docs/`。
